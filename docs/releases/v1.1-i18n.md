# v1.1 — Internationalization

> Status: Planned  
> Estimated effort: L (large — 2–4 weeks)  
> Depends on: nothing — can start immediately after v1.0.0

---

## Goal & Scope

Ship the first language beyond English. German is the initial translation target, chosen for its large self-hosting community and broad character coverage (umlauts, ß, compound nouns). The real deliverable is the i18n framework: once the infrastructure and string catalogue are in place, community contributors can add further languages via a documented PR process. RTL layout support is included in this milestone so it does not have to be retrofitted later when Arabic, Hebrew, or Persian translations arrive.

---

## Milestone Checklist

### i18n Foundation

- [x] Install and configure `next-intl` (App Router integration, server + client components)
- [x] Middleware locale detection: `Accept-Language` header, user preference override, cookie-based (no URL prefix — avoids restructuring app directory)
- [x] `messages/en.json` — canonical source-of-truth key structure, ICU MessageFormat throughout
- [ ] ICU pluralization rules verified for English: `{count, plural, one {# item} other {# items}}`
- [ ] Date/time and number formatting via `next-intl` locale-aware formatters (no raw `toLocaleString()` calls)
- [x] Locale enum added to Prisma schema (used by user preference field below)

### String Extraction

- [ ] Audit all `.tsx` components for hardcoded user-visible strings — estimated 500–800 strings
- [ ] Audit all server actions and API error messages for user-visible strings
- [ ] Replace every hardcoded string with a `t('key')` call
- [ ] Establish key naming convention: `namespace.component.purpose` (e.g., `settings.appearance.languagePicker.label`)
- [ ] No orphan keys: every key in `en.json` has at least one usage; CI check enforced
- [ ] Pluralization: all count-bearing strings use ICU plural syntax, not string concatenation

### German Translation

- [ ] `messages/de.json` — complete coverage of all keys in `en.json`
- [ ] Native-speaker review pass (no machine-translation-only strings in the release build)
- [ ] Zero English fallthrough in the German locale — every key is translated, no silent English fallback visible to users
- [ ] Locale-specific formatting validated: German date format (DD.MM.YYYY), decimal comma, thousand separator
- [ ] Right-to-left does not apply to German; layout regression tests run in `de` locale

### RTL Layout Completion

- [ ] Icon audit: all directional icons (`ChevronLeft`/`ChevronRight`, back-arrows, pagination arrows) flipped correctly under RTL using CSS `transform: scaleX(-1)` or explicit RTL variants
- [ ] Replace all physical CSS properties with logical equivalents: `margin-left` → `ms-`, `margin-right` → `me-`, `padding-left` → `ps-`, `padding-right` → `pe-`, `left`/`right` positioning → `inset-inline-start`/`inset-inline-end`
- [ ] Swipe gesture direction inverted under RTL (swipe-right = go back in LTR; swipe-left = go back in RTL)
- [ ] Arabic test locale (`ar`) added as a development-only locale to validate RTL layout without shipping an incomplete translation
- [ ] Verify sidebar, article list, reader view, settings, and modals in RTL
- [ ] Storybook (or equivalent) snapshot or Playwright screenshot for RTL layout regression

### Language Settings

- [x] `User.uiLanguage` field added to Prisma schema (string, nullable, defaults to server default)
- [ ] Migration written and tested on both PostgreSQL and SQLite
- [x] Language picker in Settings → Appearance: dropdown with language name in its own language (e.g., "Deutsch", not "German")
- [x] Preference persisted immediately via server action; page re-renders in selected locale without full reload where possible
- [x] Unauthenticated pages (login, magic link landing) respect browser `Accept-Language`

### Community Translations Workflow

- [x] `docs/contributing-translations.md` written: how to add a new locale, key naming rules, ICU syntax guide, how to run the app locally in a new locale, PR checklist
- [ ] GitHub PR template for translation PRs (language, coverage percentage, native-speaker sign-off confirmation)
- [ ] Fallback strategy documented and implemented: missing key in locale X → falls back to `en.json`, never throws; a warning is logged in development
- [ ] `scripts/check-translations.ts` (or equivalent) reports missing/extra keys per locale — runs in CI
- [ ] Translation coverage badge or table in `docs/contributing-translations.md`

### Admin Default Language

- [ ] Server Management UI: "Default UI Language" setting (applies to unauthenticated pages and new users who have not set a preference)
- [ ] Stored in existing admin settings mechanism
- [ ] Validated against the list of available (fully translated) locales — partial locales cannot be set as default

### Email Templates

- [ ] Email digest subject line and body localized to recipient's `uiLanguage`
- [ ] Transactional emails (magic link, 2FA, account deletion confirmation) localized
- [ ] Fallback to `en` if user has no language preference set
- [ ] HTML email templates use the same `messages/*.json` source files, not separate translation files

---

## Out of Scope for v1.1

- Languages other than German (framework ships; translations come via community PRs)
- Machine-translation CI pipeline (e.g., DeepL auto-suggestions in PRs)
- Per-feed language override for content display
- Translated AI summary prompts (stays English for now)
- In-app translation editor UI

---

## Dependencies / Blockers

None. Work can begin immediately on the v1.0.0 codebase.

The `next-intl` library supports Next.js App Router and is compatible with the current React 19 + Next.js 16 stack.

---

## Estimated Effort

**L — Large (2–4 weeks)**

The string extraction pass is the longest phase. German translation and RTL work can proceed in parallel once `en.json` is stable. Community workflow docs are a half-day task at the end.
